# PhysiqueOS Onboarding

## Purpose

Onboarding is not account setup.

It is the beginning of a long-term relationship between the user and PhysiqueOS.

Most products treat onboarding as a form to complete before the product can begin.

PhysiqueOS should treat onboarding as the first act of the operating system:

* understand what the user is trying to achieve
* learn what evidence already exists
* teach the user how PhysiqueOS thinks
* organize early evidence into a coherent model
* transition naturally into daily use

The objective is not to collect every possible detail on day one.

The objective is to understand enough to begin helping, while making the user feel that PhysiqueOS is already learning them.

The best first session should leave the user thinking:

> This understands what I am trying to do.

---

# Product Philosophy

Onboarding should follow the same product philosophy as the rest of PhysiqueOS.

## Goals Come First

Users do not begin with metrics.

They begin with a desired outcome.

PhysiqueOS should first understand the user's primary goal, then organize every other question, evidence source, supporting objective, and recommendation around that goal.

No onboarding question should exist merely to complete a profile.

Every question should improve understanding of the user's goal, current state, evidence quality, or next best action.

## Infer Rather Than Ask

PhysiqueOS should never ask users to manually enter information that can be inferred from existing evidence, connected sources, uploaded documents, or prior answers.

The system should ask only when:

* inference would be unreliable
* the answer materially changes interpretation
* the user should remain in control of the meaning
* the question teaches the user something useful

Good onboarding feels intelligent because it asks fewer, better questions.

## Aggregate Rather Than Duplicate

PhysiqueOS should not ask users to recreate histories they already have elsewhere.

If weight exists in Apple Health, nutrition exists in MacroFactor, scans exist in BodySpec PDFs, and workouts exist in Garmin, the product should treat those systems as evidence sources rather than competing products to replace.

Manual entry remains first-class evidence.

But onboarding should always prefer aggregation over duplication.

## Learn Over Time

The first onboarding session should not attempt to finish the model.

It should start the model.

PhysiqueOS should become more personalized as evidence accumulates, integrations connect, patterns emerge, and the user clarifies what matters.

The operating system should get smarter without becoming more complicated.

## Organize Evidence Before Offering Guidance

Early onboarding should prioritize understanding over coaching.

PhysiqueOS should first organize the user's goals, current situation, and available evidence.

Only after the system can explain what it understands should it begin offering guidance.

Guidance without understanding feels generic.

Understanding before guidance builds trust.

## Earn the Right to Coach

PhysiqueOS should not assume authority immediately.

Early language should be educational, transparent, and humble.

The product should show how it thinks, where confidence is limited, and what evidence would improve understanding.

As the user's history grows, recommendations can become more specific, more confident, and more personal.

## Respect User Autonomy

PhysiqueOS should recommend, not command.

Users remain responsible for their goals, bodies, preferences, constraints, and decisions.

The product should reduce uncertainty and surface high-leverage choices, but it should never make users feel controlled by the system.

## Every Interaction Must Matter

Every onboarding interaction should do at least one of two things:

1. improve PhysiqueOS's understanding of the user
2. help the user move toward a meaningful goal

If an interaction does neither, it probably does not belong.

# The Mental Model

Users are not building a profile.

They are building a living health model.

A profile is static.

A model changes as new evidence arrives.

PhysiqueOS should help users understand that every weight entry, scan, workout, photo, note, protocol, meal pattern, sleep trend, and recovery signal can become part of a larger picture.

Evidence accumulates over time.

The model becomes increasingly personalized.

Predictions become more accurate.

Recommendations become more relevant.

Confidence becomes more meaningful.

Onboarding should introduce this mental model without overwhelming the user.

The user should understand that PhysiqueOS is not asking questions to fill a database.

It is learning how to interpret their evidence in service of their goals.

---

# What PhysiqueOS Is

PhysiqueOS is not another tracker.

It is not simply an AI coach.

It is not a dashboard for health data.

It is a personal operating system that organizes health and fitness evidence around the user's goals.

The product exists to answer:

* Where am I headed?
* How confident are we?
* What should I do next?

Trackers collect data.

Dashboards display data.

Chatbots talk about data.

PhysiqueOS transforms evidence into understanding, confidence, and action.

Guidance emerges from understanding.

Onboarding should make this distinction clear from the beginning.

# The Onboarding Journey

Onboarding should feel like a guided conversation, not a form.

It should unfold in phases.

These phases describe the product experience and reasoning flow, not a required user interface.

## Welcome

The welcome phase introduces PhysiqueOS as an operating system for goal achievement.

The user should understand that the product is going to learn what they care about, organize evidence, and help them make better decisions.

The tone should be calm, confident, and human.

The product should not oversell.

It should explain the relationship:

* you bring goals and evidence
* PhysiqueOS organizes and interprets them
* the system becomes more useful over time

## Goal Discovery

Goal discovery identifies the user's primary outcome.

Examples:

* visible abs
* weight loss
* muscle gain
* marathon performance
* improved metabolic health
* strength maintenance
* recovery improvement

PhysiqueOS should help users express goals in outcome language rather than metric language.

The system can translate vague intent into a clearer goal model over time.

Early goal discovery should prioritize direction over precision.

## Supporting Objective Discovery

Supporting objectives explain what must remain true for the primary goal to succeed.

Users often know the headline outcome they want.

They may not know the supporting objectives that protect the quality, sustainability, or safety of that outcome.

PhysiqueOS should gently introduce supporting objectives as education, not prescription.

The system should explain why certain objectives commonly matter, then allow the user to accept, modify, ignore, or revisit them later.

## Current Situation

The current situation phase establishes where the user is starting.

This may include:

* current body weight
* recent trend
* training state
* nutrition pattern
* recovery context
* active protocols
* constraints
* known injuries
* available evidence
* upcoming milestones

The goal is not exhaustive intake.

The goal is enough context to interpret early evidence responsibly.

## Existing Evidence

Many users already have valuable evidence before joining PhysiqueOS.

Onboarding should identify what exists without requiring users to reconstruct everything manually.

Examples:

* Apple Health history
* smart scale history
* DEXA scans
* progress photos
* nutrition logs
* workout logs
* wearable data
* lab work
* PDFs
* notes
* spreadsheets

Existing evidence should be treated as a head start.

The product should make users feel that their prior effort matters.

## Evidence Discovery

Evidence discovery expands the user's understanding of what counts as evidence.

Users may think only weight, calories, and workouts matter.

PhysiqueOS should teach that context can matter too.

Examples:

* travel
* illness
* poor sleep
* stress
* hydration
* supplements
* peptides
* medication changes
* foam rolling
* soreness
* appetite
* schedule disruption
* unusual weigh-in conditions

The product should not turn every life detail into a chore.

It should help users recognize evidence when it materially improves interpretation.

## Understanding Summary

Before offering guidance, PhysiqueOS should summarize what it understands.

This is a trust-building moment.

The system should reflect back:

* the primary goal
* relevant supporting objectives
* available evidence
* missing evidence
* early confidence limits
* what the system will watch next

The summary should feel specific enough that the user recognizes themselves in it.

It should also be honest about uncertainty.

## First Daily Briefing

Onboarding should conclude with a first Daily Briefing generated from whatever evidence is currently available.

The objective is not aggressive coaching.

The objective is demonstrating understanding.

The first briefing should answer:

* what PhysiqueOS currently knows
* what evidence it used
* what remains uncertain
* what the user should do next, if anything

The ideal reaction is:

> This understands me.

## Landing on Home

After the first briefing, the user should land on Home.

Home should feel like the ongoing operating surface, not the end of onboarding.

The transition should make clear:

* onboarding started the model
* Daily Briefing explains what matters
* Home shows what to do now
* Evidence continues to improve the system

The user should know how to continue without feeling that setup is incomplete.

# Supporting Objectives

Many users focus only on their primary goal.

PhysiqueOS should help them understand the supporting objectives that commonly determine success.

Supporting objectives are not secondary goals in the traditional sense.

They are the conditions that help the primary goal become achievable, sustainable, and healthy.

Examples:

Visible abs:

* preserve lean mass
* maintain strength
* manage recovery
* collect comparable progress photos

Weight loss:

* preserve muscle
* improve nutrition consistency
* maintain activity
* reduce rebound risk

Marathon:

* manage recovery
* support nutrition
* prevent injury
* maintain training consistency

Muscle gain:

* maintain calorie surplus
* monitor body-fat gain
* track progressive overload
* protect sleep and recovery

PhysiqueOS should educate, not prescribe.

Supporting objectives should feel like helpful context, not extra obligations.

The system may suggest supporting objectives based on the user's primary goal, but users should retain control over what they care about.

Over time, supporting objectives can be promoted, demoted, retired, or added as evidence changes.

# Evidence

Evidence is any observed information that can improve PhysiqueOS's understanding of the user's trajectory.

Evidence is broader than metrics.

It includes measurements, behavior, context, documents, media, and user-reported experience.

Evidence can come from:

* voice
* photos
* wearables
* Apple Health
* Garmin
* DEXA
* blood work
* nutrition apps
* workout apps
* manual logging
* PDFs
* notes
* documents
* future integrations

Every evidence source has different reliability, freshness, relevance, and interpretation limits.

Evidence should flow through the Interpretation Layer into the Evidence Engine.

The Interpretation Layer extracts, classifies, normalizes, and compares evidence.

The Evidence Engine decides how that evidence affects understanding, confidence, goals, predictions, and recommendations.

Raw evidence should never be overwritten by computed values.

Computed values should explain evidence, not replace it.

# Existing, New, and Discovered Evidence

PhysiqueOS should distinguish between three evidence categories during onboarding.

## Existing Evidence

Existing evidence is information the user has already collected somewhere else.

Examples:

* Apple Health weight history
* BodySpec DEXA PDFs
* Garmin workouts
* Oura sleep
* MacroFactor nutrition history
* progress photos
* spreadsheets
* coach notes

Existing evidence should be imported, connected, summarized, or deferred.

The product should not make users manually duplicate it unless no better path exists.

## New Evidence

New evidence is information the user intentionally logs into PhysiqueOS.

Examples:

* morning weight
* progress photo
* protocol completion
* nutrition note
* recovery note
* DEXA upload
* training context

New evidence should be fast to enter and immediately useful.

The user should understand why logging it matters.

## Discovered Evidence

Discovered evidence is information users often do not realize is valuable.

Examples:

* foam rolling
* peptides
* recovery work
* hydration
* stress
* mood
* supplements
* travel
* illness
* unusual sleep
* unusual weigh-in conditions

PhysiqueOS should surface discovered evidence gently.

It should not ask users to track everything.

It should teach users which context matters for their specific goals and evidence patterns.

# The First Daily Briefing

The first Daily Briefing is the payoff of onboarding.

It should not feel like a generic welcome report.

It should feel like PhysiqueOS has assembled the user's available evidence into a first coherent understanding.

The first briefing should prioritize:

* what the user is trying to achieve
* what evidence is already available
* what PhysiqueOS can reasonably infer
* what remains uncertain
* what the user should do next

The tone should be:

* specific
* humble
* encouraging
* evidence-aware
* not overconfident

The first briefing should avoid pretending the system knows more than it does.

It should demonstrate the operating system's reasoning style.

The user should leave with clarity, not homework.

---

# The Future Briefing

Onboarding should also help users understand what becomes possible later.

The product may show or describe a future briefing that becomes possible after several weeks of evidence.

The purpose is expectation setting.

More relevant evidence enables:

* stronger trend interpretation
* better confidence calibration
* clearer goal progress
* more personal recommendations
* more accurate projections
* better pattern recognition
* fewer unnecessary reminders

This should not be framed as "complete setup to unlock features."

It should be framed as:

> The more relevant evidence PhysiqueOS understands, the more personally it can guide you.

The future briefing should make users excited to continue without making the first experience feel incomplete.

---

# Continued Onboarding

Onboarding does not truly end when the user reaches Home.

The first session starts the model.

The first days and weeks refine it.

PhysiqueOS should continue learning through normal use:

* evidence logs
* connected integrations
* goal changes
* priority completions
* protocol adherence
* briefing engagement
* user corrections
* missing evidence patterns

The product should avoid calling this ongoing process "onboarding" in the user experience.

To the user, it should feel like the operating system is becoming more helpful.

Continued onboarding should be opportunistic.

The system should request additional evidence only when doing so materially improves understanding.

Examples:

* "A recent progress photo would improve confidence in the visible-abs goal."
* "Your nutrition evidence is thin this week. A quick note would help explain the weight trend."
* "You mentioned travel. Should today's weight be treated as unusual context?"

The product should not turn learning into chores.

# Recommendations

Recommendations should evolve as the model becomes more personal.

## Early Recommendations

Early recommendations should use educational language.

Examples:

* "Many people pursuing similar goals benefit from tracking progress photos under consistent conditions."
* "Other PhysiqueOS users often preserve more lean mass when protein and resistance training remain visible evidence streams."
* "A DEXA scan would improve confidence, but the current trend can still be monitored without one."

Early recommendations should explain why a behavior might matter.

They should not pretend to know the user's response pattern yet.

## Later Recommendations

As evidence accumulates, recommendations should reference the user's own history.

Examples:

* "Your weight trend is most interpretable when morning weigh-ins are logged at least five days per week."
* "Your photos have been most useful when captured Friday morning under the same lighting."
* "When sleep drops below seven hours for two nights, your next-day training notes tend to show lower recovery."

The product should move from generalized education to individualized guidance.

The transition should be earned by evidence.

---

# Notifications

Notifications should never be assumed.

They should solve friction rather than create it.

PhysiqueOS should not treat notifications as a default growth or engagement mechanism.

Notifications are justified only when they help the user complete something they already value.

If users consistently complete a habit, the product may suggest reducing reminders.

If adherence struggles emerge, PhysiqueOS may suggest adding reminders.

The user should always choose.

Examples:

* "You've completed this protocol consistently for four weeks. Would you like fewer reminders?"
* "Progress photos have been missed twice. Would a Friday morning reminder help?"
* "Morning weights are consistent. I can keep reminding you or stay quiet unless the pattern changes."

Notifications should feel optional, useful, and respectful.

They should never feel like pressure.

---

# Confidence

Confidence measures PhysiqueOS's certainty.

It does not measure user performance.

This distinction is foundational.

During early onboarding, confidence should usually remain implicit.

Instead of exposing a score too early, PhysiqueOS can communicate:

* "I'm still learning."
* "I understand your goal and the evidence available so far."
* "This would become clearer with more consistent weight history."
* "DEXA would improve body-composition confidence."

Confidence becomes more valuable once the user understands what the system is evaluating.

The product should surface confidence when it can explain:

* what evidence supports the model
* what evidence is missing
* what would increase certainty
* why uncertainty remains

Confidence should create trust, not anxiety.

# Design Principles

These principles should guide every future onboarding decision.

## Never Ask What Can Be Inferred

If PhysiqueOS can reliably infer something from evidence, do not ask the user to enter it manually.

## Never Duplicate Existing Evidence

Respect the user's existing history.

Connect, import, or summarize before asking users to recreate.

## Teach Concepts Naturally

Onboarding should teach how PhysiqueOS thinks without feeling like a tutorial.

The product should explain goals, evidence, confidence, and recommendations through context.

## Expand What Counts as Evidence

Users should leave onboarding with a broader understanding of useful evidence.

This should feel empowering, not burdensome.

## Earn Permission Before Coaching

PhysiqueOS should demonstrate understanding before making strong recommendations.

## Connect Evidence Back to Goals

Every evidence request should connect to the user's goals.

If the product cannot explain why evidence matters, it should not ask for it.

## Prefer Progressive Disclosure

Do not expose the full model at once.

Reveal depth when it improves clarity.

## Keep Manual Entry Fast

Manual entry is a fallback, not the ideal future.

The long-term acquisition priority should remain:

1. API
2. passive sensor
3. voice
4. manual

## Become Smarter, Not More Complicated

As PhysiqueOS learns more, the user experience should become simpler.

More evidence should not create more work.

It should create more clarity.

# North Star

Onboarding succeeds when the user does not feel like they completed setup.

They should feel like PhysiqueOS began understanding them.

The first session should create enough clarity, trust, and momentum that the user wants to return tomorrow.

The long-term objective is simple:

> PhysiqueOS should feel less like a product asking questions and more like an operating system learning how to help.
